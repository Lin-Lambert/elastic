当输出 Markdown 格式的技术笔记、学习资料和文档时，必须严格遵循以下排版与写作规范：

## 核心原则：通俗易懂且整洁专业

绝不能只把 Markdown 当作一个"能加粗的记事本"。优秀的文档需要结构感、呼吸感和视觉焦点。哪怕是冰冷的底层逻辑，也要通过恰当的排版和"人话"变得让人有阅读欲望。

## 规则 1: 极致的"隔离感"（中英文混排与内联代码）

1. **中英文/数字盘古之白**：在中文与英文字母、中文与数字之间，**强制敲一个半角空格**。
   - ❌ `Elasticsearch有3种Node。`
   - ✅ `Elasticsearch 有 3 种 Node。`
2. **给专有名词穿上"防护服"**：变量名、专有名词、文件名或快捷命令，必须用反引号包裹成 **内联代码 (`Inline Code`)**。
   - ✅ 请进入 `elasticsearch.yml` 文件，将 `node.roles` 配置为 `[master]`。

## 规则 2: 善用引用块 (`>`) 制造"视觉题眼"

对于**核心概念、注意事项、或大白话的比喻总结**，用 `>` 引用块单独圈出来，打破大段文字沉闷的排版。

```markdown
> 💡 **KISS 核心原则 (Keep It Simple, Stupid)**
> 在 Elasticsearch 中，你可以把整个集群想象成一个"公司"，而 Node 就是公司里的一名"员工"...
```

## 规则 3: 严格的层级控制（标题与列表）

1. **H1~H3 黄金法则**：一篇文章只用一个 `#`（文档名），核心结构用 `##`（大章节）和 `###`（小节），绝不往下铺开到 H4 或 H5，再小的细分用**列表**替代。
2. **无序变有序**：如果有明确的生命周期或先后逻辑，用数字有序列表 `1. 2. 3.`；如果是平级特性展示，用无序列表 `-` 或 `*`。
3. **列表开头提词器**：在列表项的开头，尽量使用加粗的短语作为"题眼"，让读者能够一秒扫读大意。
   - ✅ **唯一决策者 (Generation)**：全世界只有 Master 有权改写路由表。

## 规则 4: 代码块带上"语言身份证号"

在使用 ` ``` ` 贴一大段配置、日志或代码时，后面必须敲上专有语言标识。
- 写 JSON 时带上 `json` 能激活花括号的彩色高亮。
- 写普通的控制台报错或返回值日志，写上 `log` 或 `text`。
- 写 YAML 配置文件，带上 `yaml`。

## 规则 5: 图文并茂的"图文组合拳"

对于复杂的系统架构、流程流转，尽量使用 `PlantUML` 画图替代长篇大论。**绝不能干巴巴地只甩一张图上去！**

1. **图前有铺垫**：在画图的代码块上方，用 1-2 句话交代清楚这张图要解决的问题或阅读动机。
2. **图中有细节**：在 PlantUML 图内部，善用 `note right` 等气泡注释，把核心公式、关键决策点标注在图上。
3. **图后有总结 (拆步解释)**：在图表结束的下方，必须紧接着用 `1. 2. 3.` 列表或"极简总结"，把图里的核心运行过程用大白话拆解再讲一遍。

## 规则 6: 目录规范 (Table of Contents)

当文档**超过 3 个二级标题（`##`）**时，必须在文档 H1 标题下方、正文上方添加目录索引，**并且必须在对应的标题处显式声明 HTML 锚点**，以彻底解决不同 Markdown 渲染器（如 IntelliJ IDEA、GitHub、Typora 等）对中文标题自动生成锚点规则不一致导致的跳转失败问题：

1. **位置**：紧跟在 `# 标题` 之后，用 `## 📑 目录` 作为标题，底部以 `---` 分割线与正文隔开。
2. **层级**：只收录 `##`（二级）和 `###`（三级）标题，不罗列更细的层级。
3. **目录格式**：使用 Markdown 锚点链接语法 `[标题文字](#锚点英文或拼音)`。**建议**锚点名称使用简短的英文拼写，避免中文字符。
4. **显式锚点注入（核心兼容方案）**：在正文对应的标题文本内部的最前面，显式插入 `<a id="锚点英文或拼音"></a>` 标签。目录中的跳转链接 `#锚点` 必须与此处定义的 `id` 完全一致。
5. **缩进**：二级标题顶格 `-`，三级标题缩进两格 `  -`。

**正确示例：**

```markdown
## 📑 目录

- [一、什么是 Node？](#what-is-node)
- [二、核心角色与分工](#node-roles)
  - [1. master（主节点）](#role-master)
  - [2. data（数据节点）](#role-data)
- [三、架构图示](#architecture)

---

## <a id="what-is-node"></a>一、什么是 Node？
(正文内容...)

## <a id="node-roles"></a>二、核心角色与分工
(正文内容...)

### <a id="role-master"></a>1. master（主节点）
(正文内容...)
```

请根据以上所有规范来输出 Markdown 文档。
